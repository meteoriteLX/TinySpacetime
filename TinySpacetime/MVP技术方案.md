# TinySpacetime MVP技术方案

## 1. 项目概述

TinySpacetime是一个基于AI的文游平台，玩家可以创建角色、携带道具穿越不同故事与世界，由AI担任主持人，支持骰子判定、状态记录、角色成长和UGC模组。

## 2. 技术栈选择

### 前端
- **框架**：Vue 3 + TypeScript
- **状态管理**：Pinia
- **HTTP客户端**：Axios
- **UI组件**：原生CSS + 自定义组件
- **构建工具**：Vite

### 后端
- **语言**：Node.js (Express)
- **AI接口**：DeepSeek API
- **数据库**：MongoDB (Atlas) [选择理由见下方分析]
- **认证**：JWT (可选，MVP阶段可简化)

### 数据库选择分析

| 特性 | SQLite | MongoDB (Atlas) |
|------|--------|----------------|
| 数据结构 | 关系型 | 文档型 |
| 存储方式 | 本地文件 | 云服务 |
| 可扩展性 | 有限 | 高 |
| 复杂度 | 低 | 中 |
| 适合场景 | 小型应用，本地开发 | 中小型应用，需要云部署 |
| 优势 | 轻量级，无需配置 | 灵活的文档结构，适合存储技能等复杂数据 |
| 劣势 | 并发性能有限，云部署不便 | 需要网络连接，有一定学习成本 |

**选择理由**：
1. **灵活性**：MongoDB的文档结构更适合存储角色技能等复杂数据，技能列表可以作为嵌套文档存储
2. **可扩展性**：随着用户量增加，MongoDB更容易扩展
3. **云部署**：MongoDB Atlas提供了方便的云服务，适合快速部署
4. **生态系统**：与Node.js集成良好，有丰富的驱动和工具
5. **数据模型**：文游平台的角色、故事、对话等数据更适合用文档型数据库存储

### 部署
- **前端**：Vercel 或 Netlify
- **后端**：Render 或 Heroku

## 4. 核心功能实现

### 4.1 角色创建
- **功能**：用户输入角色名称，分配属性和技能点数
- **属性**：力量、敏捷、智力（每个属性满值100，最高设置为80）
- **技能**：根据角色背景设定的技能（如格斗、侦查、交涉等，每个技能满值100，最高设置为80）
- **点数分配**：初始总点数有限制（参考跑团系统，如属性总点数210，技能总点数150）
- **存储**：角色信息存储到数据库

### 4.2 故事系统
- **AI集成**：接入DeepSeek API，实现用户输入→AI回复
- **对话历史**：维护对话历史，确保AI上下文理解
- **剧情推进**：AI作为主持人讲述剧情发展，兼职NPC角色在剧情中发言
- **骰子系统**：
  - 支持/roll指令，生成1-100的随机数
  - 判定规则：
    - 大成功：1-5
    - 成功：6-属性/技能值
    - 失败：属性/技能值+1-95
    - 大失败：96-100
  - AI判定：根据用户输入的行为，判定需要用到的属性或技能，然后进行骰子判定
  - 结果影响：根据判定结果修改剧情描述，影响行为是否成功

### 4.3 状态栏
- **显示内容**：当前地点、时间、事件、人际关系
- **数据来源**：从AI返回中解析或由前端维护

### 4.4 数据库设计

#### 用户表 (users)
| 字段名 | 数据类型 | 描述 |
|--------|----------|------|
| id | INTEGER | 用户ID |
| username | TEXT | 用户名 |
| created_at | TIMESTAMP | 创建时间 |

#### 角色表 (characters)
| 字段名 | 数据类型 | 描述 |
|--------|----------|------|
| id | INTEGER | 角色ID |
| user_id | INTEGER | 所属用户ID |
| name | TEXT | 角色名称 |
| strength | INTEGER | 力量属性 |
| agility | INTEGER | 敏捷属性 |
| intelligence | INTEGER | 智力属性 |
| skills | JSON | 技能列表（如{"格斗": 50, "侦查": 60, "交涉": 40}） |
| created_at | TIMESTAMP | 创建时间 |

#### 故事表 (stories)
| 字段名 | 数据类型 | 描述 |
|--------|----------|------|
| id | INTEGER | 故事ID |
| name | TEXT | 故事名称 |
| description | TEXT | 故事描述 |
| world | TEXT | 世界设定 |
| created_at | TIMESTAMP | 创建时间 |

#### 对话历史表 (dialogues)
| 字段名 | 数据类型 | 描述 |
|--------|----------|------|
| id | INTEGER | 对话ID |
| character_id | INTEGER | 角色ID |
| story_id | INTEGER | 故事ID |
| content | TEXT | 对话内容 |
| is_user | BOOLEAN | 是否用户输入 |
| timestamp | TIMESTAMP | 时间戳 |

## 5. 数据流程

1. **角色创建流程**：
   - 用户填写角色信息 → 前端验证 → 后端API → 数据库存储 → 返回角色ID

2. **游戏流程**：
   - 用户选择故事 → 前端加载故事 → 后端初始化对话 → AI生成初始剧情 → 前端展示 → 用户输入 → 后端处理 → AI生成回复 → 前端展示

3. **骰子判定流程**：
   - 用户输入/roll指令 → 前端生成随机数 → 后端处理 → AI根据结果生成描述 → 前端展示

## 8. 技术风险与应对

1. **AI API调用限制**：
   - 风险：DeepSeek API可能有调用次数限制
   - 应对：实现请求节流，优化对话历史长度

2. **数据库性能**：
   - 风险：SQLite在并发情况下可能性能不足
   - 应对：MVP阶段用户量小，后续可迁移到PostgreSQL

3. **前端响应速度**：
   - 风险：AI生成回复可能较慢
   - 应对：添加加载动画，优化用户体验
